

# План

- Я создаю систему для обучения микросервисной архитектуре. В ней 5 сервисов. Каждый сервис должен иметь в среднем 5 функций. Они не должны быть очень сложными, чтобы не разрушать наглядность, но при этом быть реалистичными и органично вписываться в систему. Сервисы должно быть удобно реализовывать через DDD, эта методика тоже является частью обучения. Предметная область - интернет магазин.
- На данный момент уже придуманы сервисы User, Product, Order.
- Мы с тобой не пишем код, мы продумываем суть сервисов и их функциональность.
- То, что проект учебный, не подразумевает никаких упрощений. Все делается как в полноценном production-решении уровня enterpise - организация взаимодействия сервисов, дополнительные библиотеки и т.д.

## User

Какие данные хранятся:

- Имя
- Номер телефона (уникальный логический идентификатор юзера в системе)
- Адреса доставки

Какие функции доступны:

- Добавить \ удалить товар в избранное
- Добавить \ удалить адрес доставки
- Задать имя
- Привязать \ отвязать email
- Привязать \ отвязать телефон
- Переключение - что использовать для входа в аккаунт - телефон или емейл. Для входа можно использовать только подтвержденный емейл или телефон.
- Уведомления в личном кабинете (например, о поступлении товара в продажу)

Идеи для бизнес-логики

- Проверка чтобы разные пользователи не могли использовать одинаковый номер телефона или email
- Опция "Автоматическое удаление товара из избранного, если он снят с продажи"
- Опция "Уведомить, если закончившийся товар из избранного снова продается"
- Выбор одного из адресов как адреса доставки по умолчанию

Дополнительно:

- Сумма выкупа
- Процент выкупа



## Product

Данные:

- Какие-то данные о товара, типа название, описание, бренд, цена, количество на складе, мб что-то еще

Функции:

- ~~Резервирование товара (на время оплаты) \ снятие резерва~~
  - Это задача сервиса Delivery
- Добавление \ удаление товара, изменение описания
- Снятие товара с продажи
  - Возможно автоматическое, когда товар закончится
- Поиск товара
  - Можно заморочиться с хитрым поиском

Дополнительно:

- История изменения цен

## Delivery

Назначение:

- Физическая реализация комплектации заказа и отправка покупателю

Функции:

- Влияние на статус заказа, продвижение статуса.
- Трек-номер
- Резерв заказа на время оплаты.



## Мысли, идеи

- Может быть отказаться от сервиса доставки? В пользу например сервиса Promotions, в котором запилить разные хитрые схемы вычисления цен?
  - Нет, нельзя. "Доставка" дает созданному заказу продолжение жизни в системе.
