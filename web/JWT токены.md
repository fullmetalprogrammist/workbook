# JWT, что значит, зачем нужен

- `JWT` - JSON Web Token
- JWT - это наиболее часто использующийся в вебе формат токена

- JWT изначально разработан как способ передачи данных о пользователе между клиентом и сервером, как альтернатива cookie и сессиям



# Базовая идея работы JWT

- Сервер создает токен, кладет в него информацию о пользователе, которая позволяет идентифицировать пользователя и понять, какие у него права доступа к системе (например, роль)
- Сервер отправляет токен клиенту, тот сохраняет его у себя и прикладывает к последующим запросам
- За счет того, что в токене лежит вся необходимая информация о пользователе и его правах, серверу не требуется ее хранить у себя
  - Это дает преимущество, например, перед сессиями, потому что упрощается горизонтальное масштабирование



# Структура JWT

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
```

- Токен - это строка, состоит из трех частей, разделенный точкой
  - Заголовок
  - Нагрузка
  - Подпись
- Как получается эта строка
  - Заголовок кодируется в Base64
  - Нагрузка тоже кодируется в Base64
  - Заголовок и нагрузка объединяются в одну строку, разделенные точкой
    - Эта строка подписывается
      - Процесс подписи - это необратимое преобразование строки через криптохэш-функцию
    - Подпись кодируется в Base64
  - Все эти три вещи объединяются в единую строку, разделяясь точкой
- P.S. Не путать кодирование, шифрование и подпись
  - Кодирование - это просто обратимое преобразование данных из одного вида в другой, для удобства передачи
  - Шифрование - это обратимое преобразование на основе ключей, чтобы враг не прочитал
  - Подпись - это необратимое преобразование

# Дополнительные детали о токенах

## Время жизни

- Обычно у токенов ограничено "время жизни", т.е. срок в течение которого токен считается валидным
- Как только срок истекает, токен считается "протухшим" и сервер отклоняет запросы с таким токеном
- Время жизни сервер кладет в нагрузку токена
  - `iat` - момент создания токена (issued at)
  - `exp` - момент окончания действия токена (expiration time)
  - Эти моменты сохранены в виде Unix-timestamp, например 1733184000

## Где хранятся токены

- На клиенте лучшим местом хранения является httpOnly-cookie
  - Потому что прочитать такую куку через JS невозможно, к ней имеет доступ только браузер, следовательно и украсть токен сложнее
  - Плюс кука автоматически прикладывается к запросу, нет нужды явно класть токен в запрос



# Архитектурные паттерны на токенах

## Access Token и Refresh Token

### Базовая идея

- Токены делятся на два типа
  - `Access Token` - используется для получения доступа к ресурсам
  - `Refresh Token` - используется для получения нового Access Token
- Такое деление призвано увеличить безопасность.
  - Access Token - "короткоживущий", например от 5 минут до часа. Поэтому если его украдут, то вор сможет пользоваться им относительно короткое время, что теоретически снижает возможный ущерб.
  - Refresh Token - "долгоживущий", например дни, недели, месяцы. Этот токен позволяет пользователю автоматически получать новый Access Token, когда тот протухает и пользователю при этом не приходится заново проходить аутентификацию.
