

# Категории SQL-операторов (DDL, DML, TCL)

- DDL
  - Data Definition Language
    - Операторы, изменяющие структуру БД и ее объектов
    - CREATE, ALTER, DROP, TRUNCATE
- DML
  - Data Manipulation Language
    - Операторы, изменяющие данные
    - INSERT, UPDATE, DELETE, MERGE
- TCL
  - Transaction Control Language
    - Операторы управления транзакциями
    - COMMIT, ROLLBACK, SAVEPOINT
- Дополнтиельно
  - DCL
    - Data Control Language
      - Управляют доступом к объектам БД
      - GRANT, REVOKE
  - DQL
    - Data Query Language
      - Оператор выбора данных
      - SELECT
      - Редко выделяют в отельную группу ("select это просто select")



# Режимы параметров процедур (IN, OUT, IN OUT)

- IN
  - "Входной" параметр
    - Дефолтный режим, явно можно не указывать
    - Можно задать дефолтное значение
    - Можно передать значение при вызове процедуры
- OUT
  - "Выходной" параметр
    - Нельзя задать дефолтное значение
    - Нельзя передать значение при вызове процедуры
    - Нужен для случаев, когда из процедуры надо вернуть значение
      - Типичный случай - вставили в процедуре новую строку в таблице, надо вернуть ее id
        - Присваиваем этот id out-параметру и все, вызывающий код получает его в переменной, которую передавал в процедуру как out-параметр
- IN OUT
  - "Входной + выходной" параметр
    - Сочетает возможности обоих
    - Используется нечасто
      - счетчики, аккумуляторы
- Особенности
  - В функциях принятно использовать только IN-параметры, а возвращать результат через RETURN
    - Все режимы работают и для функций, но это считается антипаттерном



# Процедуры и функции

- Процедуры
  - Концептуально нужны для реализации "команд" ("действий")
    - Побочные эффекты - DML (добавление, удаление, обновление )- это все идеальное место для процедур
    - Бизнес-логика тоже обычно в процедурах
    - Возвращать значение могут (через OUT-параметры)
- Функции
  - Концептуально нужны для реализации "запросов" и вычислений
    - Побочных эффектов иметь не должны
    - Могут использоваться в SQL-запросах
    - Должны возвращать единственное значение через RETURN
    - Могут использовать DML в своем теле, но это антипаттерн
      - Потому что при использовании такой функции в SQL оракл выбросит ошибку
    - Тело функции может использовать много SQL-запросов, курсоры и любую логику на PL\SQL, главное чтобы возвращала единственное значние



# Пакеты

- Пакет - это объект БД
- Концептуально - это модуль, "namespace", позволяющий объединить несколько функций, процедур, констант в единый логический блок
- Технически состоит из двух частей
  - Спецификация ("интерфейс" - описывает, какие процедуры \ функции будут "видны из пакета снаружи", их параметры)
  - Тело - реализация процедур \ функций
- Пакет создается в два шага (две команды)
  - Создается спецификация (CREATE PACKAGE)
  - Создается тело (CREATE PACKAGE BODY)
- Обновляется пакет только целиком
  - CREATE OR REPLACE PACKAGE BODY
  - Т.е. нельзя "поправить одну процедуру", надо заменять все тело пакета, просто ничего не заменяя в тех процедурах, которые менять не надо





# LOB-объекты

- LOB - Large OBjects
- Разновидности в Oracle
  - BLOB - Binary Large Object
    - Файлы, картинки, музыка, разные документы (PDF и т.д.)
  - CLOB - Character Large Object
  - NCLOB - National Character Large Object
- Любой LOB - это просто тип данных для колонки, указывается точно так же как и любой другой тип
- Способы хранения LOB
  - in-row
    - В строке таблицы хранится непосредственно значение
  - out-of-row
    - В строке таблицы хранится указатель (locator), а само значение хранится в LOB-сегменте
  - Независимо от способа хранения, внешне работа выглядит одинаково, просто под капотом оракл сам делает все как надо
  - Способ хранения оракл выбирает в момент вставки конкретных данных
    - Если соблюдаются условия (небольшой размер и т.д.), оракл сохраняет как in-row; если не соблюдаются - out-of-row
      - Но можно явно запретить in-row хранение
- Работа с LOB
  - Осуществляется через специальный пакет для работы с LOB - называется DBMS_LOB





# Партиционирование таблиц

- Партиционирование не путать с индексом
  - Индекс - ускоряет поиск среди строк
  - Партиционирование - отбрасывает часть строк, так что поиск надо вести не по всем строкам, а только по части
- Партиционирование технически - это физическая группировка строк таблицы по сегментам
  - На основе "ключа партиционирования"
  - Например, есть поле "Дата заказа", можно сделать партиционирование по году (или год+месяц). Тогда при вставке нового заказа оракл сам поймет, в какую часть таблицы физически надо вставить строку, чтобы она оказалась в нужной партиции
    - При поиске по году, соответственно, оракл тоже поймет, в какой именно партиции надо искать строку, так что часть строк даже не придется осматривать, что и ускорит выполнение запроса
  - Partition Pruning - это оптимизация, когда оракл как раз просматривает строки только из нужной партиции
  - Партиция составляется по выражению, т.е. не обязательно хранить в колонке чисто год - можно хранить полную дату, а партиционировать по году+месяцу.
  - "Удалить партицию" как "логическую" вещь нельзя, партиция - это физическая тема, и удаляется вместе с данными
    - Для изменения схемы партиционирования обычно приходится пересоздать таблицу и перелить данные
- Партиционирование делают по выражениям, которые хорошо дробят данные на крупные группы
  - Типично - год+месяц для заказов, логов и т.д.
- Виды партиционирования
  - RANGE - для ключей, хорошо поддающиеся сравнению на > <
    - Например даты, год, месяц, год+месяц и т.д.
  - LIST - для ключей, попадающих в заранее известное множество, хорошо сравнивается через =
    - Например "страна RU, US, CH", "пол муж \ жен"
  - HASH
  - COMPOSITE
