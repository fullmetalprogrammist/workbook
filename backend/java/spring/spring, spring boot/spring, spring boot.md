# Бины (beans)

## Что такое бины

- Бины - это основные "строительные блоки" в Spring-программах
- Бины представляют собой объекты, жизненный цикл которых находится в руках контейнера Spring IoC
  - Контейнер создает бины, снабжает зависимостями, инициализирует и уничтожает

## Как создать бины

- Бины создаются автоматически, их создает контейнер
- Можно явно запросить у контейнера бин
  - Но это считается в большинстве случаев антипаттерном
- Так что в основном надо просто объявить бины с помощью аннотаций, а созданием занимается спринг

## Объявление бинов

### Аннотированный класс

- Просто помечаем классы биновыми аннотациями
  - `@Component` - бин общего назначения, "нейтральный, просто бин"
    - Имеет несколько разновидностей
      - `@Service`
      - `@Repository`
      - `@Controller` \ `@RestController`
      - прочие, "тысячи их"

### Конфигурационный класс

- Комбинация: `@Configuration` на класс  + `@Bean` на метод
  - В методе создаем руками и возвращаем нужный инстанс

```java
@Configuration  // <-- 
public class AppConfig {
    @Bean  // <-- Первый какой-нибудь бин
    public UserService userService() {
        return new UserService();
    }
  
    @Bean  // <-- Другой какой-нибудь бин
    public OrderService orderService() {
        return new OrderService();
    }
}
```

- Когда бины надо создавать из библиотечных классов, этот способ является единственным решением, поскольку "чужой" класс мы не можем аннотировать непосредственно биновыми аннотациями
- Особенности
  - По умолчанию в этом подходе бины создаются со скопом singleton
    - Т.е. при каждом вызове метода возвращается один и тот же инстанс
    - Если надо чтобы метод возвращал каждый раз новый инстанс, надо дополнительно добавить методу аннотацию `@Scope("prototype")`
      - Но это считается антипаттерном. Исторически @Configuration-класс используется для объявления бинов, которые должны существовать именно в единственном экземпляре на все приложение

## Жизненный цикл бинов

- `Жизненный цикл` - это процесс из нескольких этапов, которые бин проходит от момента создания до момента уничтожения
- Основные этапы такие
  - Создание бина
  - Внедрение в него зависимостей
  - Инициализация (выполнение метода бина, аннотированного через `@PostConstruct`)
  - Жизнь-работа
  - Освобождение ресурсов (выполнение метода бина, аннотированного через `@PreDestroy`)
  - Уничтожение бина

## Скоуп бинов (scope)

- Адекватного русского устоявшегося перевода нет. "Область видимости" не отражает суть
- Суть скоупа это "время жизни + количество экземпляров"
- Скоупы делятся на категории
  - `core` - так сказать, скоупы обычного спринг-приложения
    - `singleton` - бин существует один на все приложение, контейнер при требовании такого бина создает его, кэширует, а потом каждый раз возвращает один и тот же инстанс
    - `prototype` - контейнер каждый раз создает новый инстанс бина
  - `web` - скоупы для спринговых веб-приложений (Spring MVC)
    - `request` - на каждый HTTP-запрос создается новый инстанс бина
    - `session`
    - `application`
    - `websocket`
  - прочие - в других типах спринг-проектов могут быть свои скоупы
- Специализированные скоупы дополняют базовые, т.е. в веб-приложениях доступны core-скоупы + собственные веб-скоупы
- Скоуп можно задать явно через аннотацию `@Scope("тут-тип-скоупа")` в дополнение к биновой аннотации



# Spring IoC

- `IoC` - Inversion of Control, принцип согласно которому задачу создания объектов мы делегируем какой-то абстракции
  - Т.е. не сами создаем объекты (через new например), а их за нас создает какой-то специальный инструмент
- В Spring эту задачу выполняет `ApplicationContext`
  - Он ищет классы, аннотированные биновыми аннотациями, регистрирует их у себя и когда где-то требуется бин, он его создает и внедряет



# Dependency Injection

- В спринге три способа внедрения зависимостей
  - Через конструктор
  - Через сеттеры
  - Через поле
- Все они используют аннотацию `@Autowired` для маркировки, что требуется автоматическое внедрение зависимости
  - Аннотация вешается на соответствующий член класса - конструктор, метод-сеттер или на поле
- Предпочтительным считается внедрение через конструктор
  - Потому что зависимость после внедрения нельзя перезаписать, следовательно не сломаешь поведение
  - Зависимости через конструктор обязательные, стало быть явно видно что нужно классу для работы
  - Зависимости проще замокать в тестах

## Конструктор

```java
@Component
public class UserService {
    private final OrderService orderService;

    @Autowired // Вешаем @Autowired на конструктор
    public UserService(OrderService orderService) {
        this.orderService = orderService;
    }
}
```

- Если конструктор один, то @Autowired можно не вешать
- Если конструкторов несколько, тогда один из них явно надо пометить @Autowired, иначе ошибка будет

## Сеттеры

```java
@Component
public class UserService {
    private OrderService orderService;

    @Autowired  // Вешаем @Autowired на конструктор
    public void setOrderService(OrderService orderService) {
        this.orderService = orderService;
    }
}
```

## Поле

```java
@Component
public class UserService {
    @Autowired  // Вешаем @Autowired на поле
    private OrderService orderService;
}
```





# Прокси

## Зачем нужны прокси

- Концептуально под прокси понимается обертка над исходным объектом
  - Клиент работает с оберткой, при этом обертка выглядит для него как оригинальный объект, т.е. имеет все те же методы
- Прокси позволяют дополнить методы оригинального объекта дополнительными действиями ("обвязка"), без модификации кода оригинальных методов
  - Как правило это что-то инфраструктурное, вроде логирования, транзакционности, метрики \ профилирование и т.д.
    - Например, перед вызовом оригинального метода запустить таймер, потом вызвать метод, а потом залогировать значение таймера, чтобы понять как долго выполнялся метод - это хороший пример применения прокси
  - Действия для всех методов обычно одинаковые
    - Обычно вопрос в том, к каким методам их применять, а к каким нет, а не "для метода foo действия A и B, а для метода bar - действия Z, X"
  - Итого - прокси имеет все такие же методы как оригинальный объект и при вызове этих методов срабатывает обвязка + выполняется метод оригинального объекта
- Через прокси реализуется аспектно-ориентированное программирование (AOP)

## Как технически создаются прокси

- В Spring есть интерфейс `BeanPostProcessor`
  - У него два метода
    - `postProcessBeforeInitialization` - выполняется перед @PostConstruct
    - `postProcessAfterInitialization` - выполняется после @PostConstruct
    - каждый из этих методов возвращает объект
      - можно вернуть оригинальный бин, который пришел в метод
      - а можно вернуть другой объект - это и будет прокси, который попадает в контейнер вместо оригинального бина
        - обычно прокси возвращают из post-метода, потому что к этому моменту бин уже прошел инициализацию и готов к работе
  - before-метод используется реже after-метода
    - да и в целом оба метода обычно используются разработчиками библиотек, нежели прикладными программистами
  - Интерфейс старый, но почти не менялся за 20 лет, относится к самой базе
  - От @PostConstruct отличается тем, что @PostConstruct - это работа с одним конкретным бином и там по сути возможности ограничены только какой-то настройкой бина. А BeanPostProcessor срабатывает на все бины и позволяет по сути заменить бин другим объектом (проксей)

Примерно реализация выглядит так:

```java
@Component
public class ProxyBeanPostProcessor implements BeanPostProcessor {

    @Override
    public Object postProcessAfterInitialization(Object bean, String name) {
        if (!(bean instanceof Foobar)) return bean;

        return Proxy.newProxyInstance(
            bean.getClass().getClassLoader(),
            bean.getClass().getInterfaces(),
            (proxy, method, args) -> {

                // 1. если есть аннотация → логируем
                if (method.isAnnotationPresent(Loggable.class)) {
                    System.out.println("LOG before " + method.getName());
                    Object result = method.invoke(bean, args);
                    System.out.println("LOG after " + method.getName());
                    return result;
                }

                // 2. если метод bar → вообще запрещаем
                if (method.getName().equals("bar")) {
                    throw new RuntimeException("bar is forbidden");
                }

                // 3. иначе — просто вызываем
                return method.invoke(bean, args);
            }
        );
    }
}
```

- `Proxy.newProxyInstance` - для нашего удобства (механизм Java), чтобы спринг сам создал обертку для оригинального класса, а нам через лямбду осталось только осталось написать обвязку для методов
  - Весь код лямбды будет выполняться перед вызовом каждого метода





# AOP (Aspect-Oriented Programming)

- `AOP` - это парадигма программирования
  - Код обычно делится на бизнес-логику и сквозную логику, т.е. дополнительную, инфраструктурную (метрики, логирование, транзации, обработка исключений и т.д.)
  - AOP парадигма подразумевает выделение такой сквозной логики в отдельные классы \ модули, которые называют `аспектами`
  - Т.о. аспекты реализуется отдельно от бизнес-логики, не зашумляют ее, а затем применяются к ней с помощью средств языка \ фреймворка
- В джаве AOP реализуется через прокси

## aspect, advice, pointcut, join point, introduction

- Это специфичные для джавы понятия. В других системах мб что-то похожее, но в общем именно эти формулировки - джавовские. Но вся эта история изначально пошла из джавы, поэтому остальные во многом эти идеи переняли.
- `join point` - это точка кода, к которой можно присоединить сквозную логику
  - пример таких точек
    - создание объекта
    - изменение поля
    - вызов метода
  - но в спринге по сути это всегда именно вызов метода
- `pointcut` - правило, которое определяет на какие join point'ы применять сквозную логику
  - реализуется через аннотацию `@Pointcut` в которую передается выражение вроде регулярки (язык AspectJ)
- `advice` - это "совет", в какой момент времени по отношению к join point выполнять сквозную функциональность
  - выражается несколькими аннотациями, вроде `@Before`, `@After`, `@Around` и т.д.
    - например, перед точкой ("перед выполнением метода"), после точки ("после выполнения метода"), "до и после (around) выполнения метода" и прочие
- `aspect` - это класс, в котором реализована сквозная функциональность
  - Тут же описывается как она привязывается к join point'ам (через pointcut'ы и advice'ы)
  - каждый аспект реализуется отдельным классом, аннотированным через `@Aspect`
  - все остальные аннотации навешиваются на методы внутри этого класса

Пример:

```java
@Service
class UserService {  // Бизнес-логика

    public void createUser(String name) {
        System.out.println("Creating user: " + name);
    }

    public void deleteUser(String name) {
        System.out.println("Deleting user: " + name);
    }
}

@Aspect
@Component
class LoggingAspect {  // Сквозная функциональность

    @Pointcut("execution(* UserService.*(..))")
    public void userMethods() {}

    @Before("userMethods()")
    public void before() {
        System.out.println("BEFORE");
    }

    @After("userMethods()")
    public void after() {
        System.out.println("AFTER");
    }

    @Around("userMethods()")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        System.out.println("AROUND BEFORE");
        Object result = pjp.proceed();
        System.out.println("AROUND AFTER");
        return result;
    }
}
```

- P.S. Для ознакомления. Фактически пишется от руки редко, в основном AOP используется через готовые библиотечные аннотации

## Как @Aspect связан с прокси

- Спринг находит классы, помеченные через `@Aspect`
  - По поинткатам понимает, на какие бины должен работать аспект и когда эти бины создаются, он делает для них прокси



# Транзакции

## @Transactional

- В спринге есть встроенные средства для упрощения работы с транзакциями
- Главная аннотация `@Transactional`
  - Можно аннотировать через нее метод, в котором идет работа с БД
    - Если в методе происходит исключение, транзакция автоматически откатывается, иначе - автоматически коммитится
  - Работает независимо от конкретной СУБД, в своем коде опирается на отвязанные от СУБД вещи, которые для конкретных СУБД реализует `PlatformTransactionManager`
- Как работает
  - Когда мы применяем @Transactional к какому-то методу \ методам, то спринг создает прокси для бина
  - При вызове @Transactional-метода на прокси срабатывает транзакционная обвязка

## Проблемы с транзакциями

### @Transactional и self-invocation

- `self-invocation` - это когда метод класса вызывает внутри себя другой метод этого же класса
- Суть проблемы
  - Если в каком-то методе класса вызвать другой метод, помеченный через @Transactional, то транзакционная обвязка не сработает
- Причина проблемы
  - Когда метод бина вызывает другой метод, то он вызывается на оригинальном бине, а не на проксе
  - А обвязка срабатывает только при вызове метода на проксе
  - Поэтому при self-invocation обвязка не срабатывает
  - Итого: вызываем метод на проксе > срабатывает обвязка + вызывается метод оригинального бина > в этом методе вызывается другой метод > это очевидно уже происходит на оригинальном бине, а не на проксе => обвязка не срабатывает
- Как решать проблему
  - Избежать вызов через this
    - По сути, самый ходовой способ - вынести транзакционную логику в отдельный класс, внедрить в оригинальный класс этот класс с транзакциями и работать через него
