# Что такое докер

- докер - это платформа для создания и запуска контейнеров
  - технически докер - это несколько независимых процессов в ОС, которые работают вместе для достижения целей
- докер - не полностью самостоятельная технология
  - он базируется на механизмах ОС линукс
  - поэтому он может работать только на линуксе, либо на других ОС, которые могут линукс эмулировать (например технология WSL на винде)



# Что такое образ и контейнер

## Концептуально

- Образ - это неизменяемый "дистрибутив", в котором находится программа и все ее зависимости (библиотеки, настройки и т.д.)
- При запуске образа создается "контейнер". 
- Контейнер можно запускать повторно и между запусками он сохраняет свое состояние.
  - Например, можно добавить в БД данные, остановить контейнер, запустить снова, и данные с прошлой работы там будут.



## Технически

- базовые механизмы ядра линукса, на которых строится "контейнеризация"
  - `cgroups` - механизм ограничения ресурсов для процесса (cpu, память, сеть и т.д.)
    - процесс либо не получает больше ресурсов, чем ему положено, либо в каких-то ситуациях ОС просто может его убить
  - `namespace` - механизм ограничения видимости для процесса
    - видимость можно ограничивать для разных вещей
      - например на другие процессы, на точки монтирования
        - по умолчанию, когда ограничений нет, процесс видит все другие процессы и всю файловую систему. Но если поместить его в неймспейсы с этими типами ограничений, то он например не будет видеть другие процессы и будет видеть не всю файловую систему, а только фрагмент
      - и т.д. - есть больше видов ограничений видимости, но это не важно, важна только суть - ограниченная видимость для процесса как бы помещает его в изолированное пространство, за пределами которого он ничего не видит и соответственно не может влиять на это
  - "группы" и "неймспейсы" - это метаинформация, привязанная к процессу, на основе которой ядро ориентируется, что давать \ не давать \ показывать \ не показывать процессу
- `контейнер` - это процесс, для которого заданы ограничения ресурсов и ограничение видимости
  - фактически роль докера при "поднятии контейнера из образа" - это дать ядру ОС команду запустить процесс с ограничениями
    - этот процесс будет - то что мы пишем в CMD для образа
- контейнер и неймспейсы
  - контейнер находится одновременно в нескольких неймспейсах разных типов, но всегда только в одном неймспейсе конкретного типа
    - например
      - все контейнеры находятся в разных pid-неймспейсах, за счет чего не видят друг друга
      - но несколько контейнеров могут находиться в одном net-неймспейсе, за счет чего они как бы оказываются в одной сети и могут обмениваться информацией





# Полезные факты

## Ментальная картина запуска контейнера из образа

- ментальная картина запуска контейнера из образа
  - образ представлен архивом
  - образ можно визуализировать себе как самостоятельную файловую систему (как например флешку)
  - в ней хранятся бинарники программ и их зависимостей
    - грубо говоря, в образе postgres будет бинарник постгри и всех ее зависимостей
    - в образе с jvm будет бинарник этой jvm
    - и т.д. для всех остальных случаев
  - "поднять контейнер из образа" фактически означает следующее (грубо и приблизительно)
    - ОС монтирует файловую систему образа к себе в какую-то точку
    - на хостовой ОС запускается процесс ("контейнер")
      - мы поднимаем контейнер командой `docker run имя-образа`, там через цепочку вызовов "докерских демонов" в итоге запускается команда из CMD, которую мы написали при создании образа (например, `CMD java -jar product-0.0.1-SNAPSHOT.jar`) и так и получается процесс
    - ОС ставит эту смонтированную точку как корень для процесса контейнера
      - т.о. для контейнера корнем файловой системы становится эта точка, а не реальный корень хостовой ОС

## Один контейнер - один процесс

- best practice в контейнеризации является когда при "поднятии контейнера" создается один процесс (главный), и от него если надо создаются дочерние процессы
- когда создаются два и более независимых процесса, не связанных общим родителем - это плохо, потому что тогда управлять ими становится невозможно
  - когда мы захотим остановить контейнер (читай "убить процесс"), то убьем только главный, а остальные останутся работать
- если сильно надо - можно использовать `супервизор`, который объединит под собой несколько процессов, став их родителем

## Одинаковые порты у разных контейнеров

- вопрос: если сделать несколько приложений, которые слушают одинаковый порт (8080 например), упаковать их в образы и запустить через докер, то не будет ли конфликта портов?
  - нет, не будет. Каждый процесс запускается в своем net-namespace, поэтому порт 8080 будет локальным для каждого процесса ("контейнера")
  - конфликт будет, если при пробрасывании портов указать одинаковый порт для разных процессов



# Термины

- `демон` - процесс, выполняющийся в фоновом режиме, без привязки к терминалу или пользовательской сессии
  - демон - это название из юникса \ линукса, в винде это называется `служба`
  - демоны запускаются при старте ОС, еще до того как мы залогинились в систему, и остаются работать, когда мы разлогиниваемся
- `бинарник` - скомпилированный в машинный код файл программы, готовый к выполнению
  - применимо только к файлам, непосредственно выполняемым операционкой. Например, .jar назвать бинарником нельзя, потому что он выполняется виртуальной машиной (jvm). Вот сама jvm - это уже бинарник
- `Процессы` и `программы`
  - Программа - файл на диске, который можно выполнить
  - Процесс - запущенная программа
    - Процесс имеет
      - pid - собственный идентификатор процесса
      - ppid - parent pid, идентификатор процесса, который его запустил
      - дополнительные фишки
        - виртуальное адресное пространство - каждый процесс думает что он располагает всей памятью компьютера, хотя на деле работает с виртуальными адресами. Эти виртуальные адреса в реальные переводит ОС. Т.о. процесс физически не может залезть в память другого процесса и что-то там повредить \ прочитать.
