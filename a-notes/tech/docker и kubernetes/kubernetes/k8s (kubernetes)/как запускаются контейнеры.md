# Образ и контейнер

- ментальная картина запуска контейнера из образа
  - образ представлен архивом
  - образ можно визуализировать себе как самостоятельную файловую систему (как например флешку)
  - в ней хранятся бинарники программ и их зависимостей
    - грубо говоря, в образе postgres будет бинарник постгри и всех ее зависимостей
    - в образе с jvm будет бинарник этой jvm
    - и т.д. для всех остальных случаев
  - "поднять контейнер из образа" фактически означает следующее
    - хостовая ОС запускает процесс - это фактически и есть то, чем реально является контейнер - просто процесс на хостовой ОС
      - мы поднимаем контейнер командой `docker run имя-образа`, там через цепочку вызовов "докерских демонов" в итоге запускается команда `CMD java -jar product-0.0.1-SNAPSHOT.jar` из образа, и вот так получается процесс контейнера
    - ОС монтирует файловую систему образа к себе в какую-то точку
    - ставит эту точку как корень для процесса контейнера
      - т.о. для контейнера корнем файловой системы становится эта точка, а не реальный корень хостовой ОС
  - дополнительные факты
    - каждый процесс-контейнер запускается в отдельном namespace, это значит что процесс не видит кроме себя никакие другие процессы. Т.о. контейнеры изолируются друг от друга и от процессов ОС
- best practice по использованию
  - в одном контейнере надо запускать только один процесс
    - например, только java-прогу, но не java-прогу + postgres
      - почему - разбираться впадлу сейчас, но связано это вроде с тем, что только один процесс будет главным, а остальные - фоновыми и управление ими душное и непрозрачное





# Термины

- `демон` - процесс, выполняющийся в фоновом режиме, без привязки к терминалу или пользовательской сессии
  - демон - это название из юникса \ линукса, в винде это называется `служба`
  - демоны запускаются при старте ОС, еще до того как мы залогинились в систему, и остаются работать, когда мы разлогиниваемся
- `бинарник` - скомпилированный в машинный код файл программы, готовый к выполнению
  - применимо только к файлам, непосредственно выполняемым операционкой. Например, .jar назвать бинарником нельзя, потому что он выполняется виртуальной машиной (jvm). Вот сама jvm - это уже бинарник
- `Процессы` и `программы`
  - Программа - файл на диске, который можно выполнить
  - Процесс - запущенная программа
    - Процесс имеет
      - pid - собственный идентификатор процесса
      - ppid - parent pid, идентификатор процесса, который его запустил
      - дополнительные фишки
        - виртуальное адресное пространство - каждый процесс думает что он располагает всей памятью компьютера, хотя на деле работает с виртуальными адресами. Эти виртуальные адреса в реальные переводит ОС. Т.о. процесс физически не может залезть в память другого процесса и что-то там повредить \ прочитать.
