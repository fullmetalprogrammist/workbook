# Запуск команд

- все выполняется из PowerShell под администратором
- `minikube` - команда управления непосредственно кластером (создание и запуск, остановка и т.д.)



# Profile и кластер

- profile - это файл с настройками для кластера (драйвер, память, cpu и т.д.), по ним кубер поднимает кластер
  - профиль создается при старте кластера, настройки можно задать при старте или изменить потом
  - сначала создается профиль, потом по нему поднимается кластер
- в кубере profile и кластер соотносятся 1:1, т.е. профиль описывает только один кластер

# Работа с кластером (minikube)

## Создание и запуск кластера

- создание сопряжено с запуском, отдельно создать, не запуская - нельзя

```
minikube start -p my-cluster -d hyperv
```

- `-p` - имя кластера
  - без -p имя по умолчанию будет minikube
- `-d` - драйвер, на котором будет работать кластер
  - `hyperv` - нативный виндовый
  - `docker` - драйвер докера
    - чтобы использовать, докер должен быть запущен
    - более легковесный чем hyperv
  - без -d кубер выберет дефолтный для системы драйвер (но как будто явно указать лучше)

Дополнительно:

- одновременно можно запускать несколько кластеров, но на практике для локальной разработки это обычно бесполезно



## Список кластеров

```
minikube profile list
```

- показывает таблицу всех кластеров
  - работающие сейчас выделены зеленым
  - колонка `profile` - имя кластера
  - колонка `active profile` - звездочкой отмечен активный кластер (на него уходят действия от команды minikube)
  - колонка `active kubecontext` - звездочкой отмечен активный контекст (на него уходят действия от команды kubectl)



## Остановить кластер

```
minikube stop -p my-cluster
```



## Удалить кластер

```
minikube delete -p my-cluster
```

- `-p` - удалить один конкретный кластер
- `--all` - удалить вообще все кластеры
- удалить несколько конкретных одной командой нельзя, можно либо все вообще, либо по  одному



## Какой кластер сейчас активный

```
minikube profile list
```

- смотреть где стоит звездочка в колонке active profile
- что дает "активность"
  - все minikube-команды без флага -p срабатывают на активный кластер
  - активность сохраняется в конфигах кубера и после перезагрузки, например, активным остается тот же кластер, что и был



## Сделать указанный кластер активным

```
minikube profile my-cluster
```

- если сделать кластер активным, то на него будут работать команды minikube, если не указывать флаг -p



# Дополнительные команды

## Дашборд управления ресурсами в кластере

```
minikube dashboard
```

- `--url` - просто вывести в консоль url, по которому станет доступен дашборд, без автоматического открытия браузера
- `--port` - вручную задать желаемый порт. Без этого выбирается случайный свободный порт.
- особенности
  - дашборд блокирует терминал, для выхода `Ctrl + C`

Что такое дашборд:

- дашборд это UI для управления ресурсами кластера кубера (поды, деплоймнты, сервисы и т.д.)
- с его помощью нельзя управлять самим кластером (остановить его, удалить и т.д.)



## Импорт docker-образа в minikube

- чтобы пользоваться докер-образом надо сначала загрузить его в виртуалку, на которой работает кубер
  - это не зависит от драйвера, надо и при hyperv, и при docker
- импортируем
  - `docker images` - чтобы посмотреть список образов докера и узнать точное имя нашего
  - `minikube image load foobar:latest` - загружаем образ в виртуалку

### Список образов в виртуалке

```
minikube image list
```

### Удалить образ из виртуалки

```
minikube image rm foobar:latest
```
