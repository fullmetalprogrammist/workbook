# Deployment

- деплоймент - это объект кубера
  - изначально представляет собой yaml-файл
  - после применения создается объект деплоймента
- деплоймент описывает структуру одного пода
  - 
- , в котором описано, какие контейнеры надо развернуть, сколько реплик и т.д.
- зачем нужен
  - управление репликами сервиса (сколько их должно быть)
    - автоматически поднимает их, если они упали
  - обновление сервиса на другую версию образа
    - при этом перезапускается весь под, а не только тот контейнер, образ для которого обновили



# Настройки деплоймента

```yaml
# деплоймент
apiVersion: apps/v1
kind: Deployment
metadata:
  name: product
spec:
  replicas: 1
  selector:
    matchLabels:
      app: product
  template:
    metadata:
      labels:
        app: product
    spec:
      containers:
        - name: product
          image: ms-product
          imagePullPolicy: Never
          ports:
            - name: foobar
              containerPort: 8081
            # или просто
            # - containerPort: 8081
```

- `spec`
  - `replicas` - сколько подов должен поднять этот деплоймент
  - `selector` - селектор нужен, чтобы деплоймент понимал, за какие поды он отвечает. Т.е. когда поды подняты и работают, у них есть метки (задаются при поднятии), и селектор благодаря им понимает "ага, вот за этот под я отвечаю".
    - `matchLabels` - селектор работает через метки
      - `app: product` - `имя метки: значение метки`
        - метки, которые мы указываем здесь, должны совпадать с метками, которые мы укажем далее в разделе `template`
        - меток может быть больше, они комбинируются по принципу "И", т.е. у пода должны быть все указанные метки, чтобы деплоймент считал его своим
  - `template` - шаблон, описывающий под
    - `metadata`
      - `labels` - метки для поднимаемого пода. Когда деплоймент поднимает под, он присваивает ему указанные тут метки. Метки используются потом селектором, чтобы деплоймент понимал, за какие поды он отвечает. Поэтому важно метки в обоих разделах сделать одинаковыми, не ошибиться
        - `app: product`
    - `spec` - описание пода
      - `containers` - какие контейнеры будут крутиться в поде
        - `name` - имя для контейнера
        - `image` - из какого образа поднять контейнер
        - `imagePullPolicy` - TODO
        - `ports` - раздел для указания портов, которые слушает приложение в контейнере.
          - технически, раздел не обязателен, т.е. если не указать порты явно, на них все равно можно ссылаться вручную, например, в service. Однако явное указание нагляднее, плюс можно задать например имя для порта и ссылаться на него по имени, а не по числу
          - `containerPort` - цифра порта
          - `name` - можно дать порту буквенное имя
