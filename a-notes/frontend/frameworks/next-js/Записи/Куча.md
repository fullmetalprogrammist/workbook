# Виды роутинга

- Pages Router - старая версия.
- App Router - современная версия.

Оба относятся к "файловому" роутингу, т.е. опираются на структуру приложения в файловой системе.





# Виды рендеринга

Концептуально рендеринг может быть:

- Серверный (SSR - Server Side Rendering)
  - html генерируются на лету, когда клиент обращается по url. Приложение выполняет функцию, например по добыче данных из БД, эти данные передаются в компонент, который должен их отображать, получается итоговый html и он уходит в браузер клиенту, а потом догружается js-чанк. Т.о. SSR подразумевает, что при первом обращении к странице происходит рендеринг на сервере, а дальше страница работает как обычное SPA. Т.е. чтобы например отрендерить новые данные, браузер шлет запрос за данными, а не за полностью новым html.
- Статический (SSG - Static Site Generation)
  - html генерируется при сборке приложения и становится одним из файлов приложения и хранится на диске. При сборке можно выполнить функцию по добыче данных и эти данные "прожгутся" в страницу. Заменить их можно будет только если повторно сделать сборку приложения. Когда клиент обращается по url к такой странице, приложение отдает ее мгновенное как любой статический файл, а потом догружается js-чанк и страница становится как обычное SPA.
- Клиентский (CSR - Client Side Rendering)
  - Клиент получает пустой html и js и весь рендеринг происходит программно в браузере. Т.е. на сервере никакого html не генерируется.
- Гибридный (SSG + CSR)
  - Часть страницы является статической (например, хедер, футер, меню), а часть - динамической. При сборке страница рендерится в html и сохраняется на диск. Когда клиент запрашивает такой url, то получает этот первичный html + js для гидрации. Первичный html он видит сразу, без данных, с надписью Loading (например), а потом полученный скрипт загружает данные и они встают на свое место.

И pages, и app-роутеры поддерживают все эти виды рендеринга.

# Роуты-"страницы" и роуты-"api"

В Next-приложении роуты концептуально могут быть двух типов:

- "Страницы" - их результатом является html. Они нужны, чтобы отдать клиенту html + js.
- "API" - их результатом являются данные. Это как бы бэкенд. Если например нам надо прочитать данные из БД и использовать для этого секреты (логин, пароль и т.д.), то мы делаем это в api-роутах. Они выполняются на сервере и в браузер клиенту не попадают, т.о. секреты остаются в безопасности.



# Pages Router

В App Router функции getServerSideProps, getStaticProps и getStaticPaths не существуют. Это концепция, специфичная для Pages Router.

Сценарии рендеринга:

- SSR - клиент запросил страницу, приложение выполнило серверную функцию, отдало данные компоненту, приготовило готовый html и отдало его клиенту. Затем он запросил JS (гидратация).
  - Для динамичного контента, который должен быть актуален при каждом запросе, +возможно требуется персонализация. Пример: лента новостей.
- SSG - при сборке приложения подготовились готовые html и сохранились на диск. Клиент запросил страницу, получил сразу готовый html. Затем запросил JS.
  - Для стабильного не меняющегося контента. Пример: информационные страницы со статической информацией.
- SSG с динамическими путями - как SSG, только готовится много статических однотипных страниц для каждого возможного пути, который вычисляется по заданному шаблону. Например, products/1, products/2 и т.д.
  - Много однотипных статических страниц с однотипным URL. Пример: статьи в блоге, какой-нибудь каталог товаров, который редко обновляется (ИП Залупина с каталогом бесполезных БАД'ов)
- СSR - ничего готового нет. Клиент получает пустой html и JS. Все формирование html происходит на клиенте.
  - "Программа в браузере, а не страница в интернете", когда SEO не важно. Пример: почта, приложение банка.
- Гибрид (SSR\SSG + CSR) - когда на сервере подготавливается часть данных, клиенту отправляется html с этими данными, а после гидрации догружаются остальные данные.

В архитектуре Pages Router всегда есть гидратация, т.е. механизм выглядит так:

- Браузер обращается к странице приложения.
- Приложение отдает ему html.
- Браузер показывает html и загружает js.
- Происходит гидратация - загруженный js прицепляется к html и получается интерактивность приложения.

Т.е. к html всегда в догонку уходит js. В архитектуре App Router гидратация не обязательна, приложение может отдать просто html без js.

# База работы Next приложения, сравнение с SPA

Здесь надо править по мере улучшения понимания. За раз разобраться в том, как реально все устроено оказалось невозможно. Тут описаны какие-то базовые правильные вещи, но сами процессы могут включать нюансы. Так что "здесь все правильно написано в целом, но есть нюанс..."

## Основы

Как работает SPA

- После сборки SPA приложение представляет собой единственный html-файл и один js-бандл с логикой всех "страниц" приложения.
- Когда браузер загружает SPA приложение, он получает с сервера html и js и по сути все - больше ему не требуется обращаться на сервер, т.к. вся функциональность целого приложения у него уже есть. На сервер он делает только запросы за данными. Т.е. когда мы "переходим между страницами" SPA приложения, то это происходит на едином html, в котором просто меняется DOM с помощью js.
- При сборке SPA есть возможность разбить единый бандл на "чанки" - функциональность каждой "страницы" будет находиться в отдельном небольшом js-файле. Полезно для оптимизации при больших размерах единого бандла. И когда пользователь захочет "перейти на другую страницу", то браузер запросит с сервера js-файл этой страницы. Важно то, что он получит только js, т.е. html-страница останется та же самая, просто он получит дополнительный скрипт с функциональностью нужной "страницы".
- Важно то, что изначально браузер получает с сервера *пустой* html, и выстраивает DOM программно. Это негативно сказывается на SEO и роботах, потому что они не имеют никакой информации о странице и ее содержимом, т.к. все это заполняется программно на клиенте.

Как работает Next:

- Что получится при сборке Next-приложения зависит от настроек стратегии рендеринга конкретных "страниц".
  - SSG - каждая "страница" приложения, для которой выбрана SSG-стратегия, превращается в отдельный html + js. Они хранятся в файловой системе сервера, где крутится приложение.
  - SSR - если для "страницы" выбрана SSR-стратегия, html генерируется не при сборке приложения, а когда на сервер приходит запрос этой страницы. Сервер подготавливает на лету html и возвращает его с js-бандлом клиенту. Этот html не хранится в файловой системе.
- Ключевое здесь, что эти html - не пустые, а полноценные страницы с контентом, что полезно для SEO и роботов.
- Поскольку сервер отдает уже сформированный html, то первая загрузка для клиента выглядит быстрее.
- Дальше все переходы внутри приложения работают как в обычном SPA. При попытке перейти, например, со страницы Main на About, приложение перестраивает текущий html.

Итог:

- При первом запросе любой страницы приложения клиент получает уже готовый html, который надо просто отобразить.
- При последующих переходах между страницами внутри приложения Next работает как классическое чанкированное SPA, запрашивая с сервера чанк нужной страницы и перестраивая имеющийся html.

## Детали

- Prefetching - переходы внутри Next-приложения организуют с помощью next/link. Когда во вьюпорте появляется такой элемент Link, клиент автоматически отправляет на сервер запрос для подгрузки js-чанка страницы, на которую ведет такой Link. Поэтому если пользователь решил перейти, для него переход будет мгновенным, т.к. чанк уже будет загружен.
- Hydration - при запросе страницы сервер отдает html, который браузер немедленно показывает и параллельно начинает загружать js-бандл. Т.е. страница становится интерактивной не мгновенно, а только когда скрипт загрузится.
