# Стратегии рендеринга страницы

## Виды рендеринга

- [ ] Базовые стратегии
  - [ ] SSR - Server Side Rendering - Рендеринг на стороне сервера
  - [ ] SSG - Static Site Generation - Статическая генерация сайта
  - [ ] ISR - Incremental Static Regeneration - Инкрементальная статическая регенерация
  - [ ] CSR - Client Side Rendering - Рендеринг на стороне клиента
  - [ ] Для каждой страницы можно выбрать стратегию индивидуально
- [ ] RSC - React Server Component - Архитектура, которая позволяет на одной странице использовать комбинацию других стратегий, для каждого компонента - наиболее подходящая ему стратегия.

- [ ] Когда использовать Next js, а когда обычный React? 
- [ ] Ваше описание идеально для старого Pages Router. Новый App Router (на базе Server Components) вносит adjustments:
  - **Рендеринг по умолчанию:** Компоненты в `app/` рендерятся **на сервере** (как SSR, но без хуков состояния). Это не просто HTML, а специальный поток данных.
  - **Клиентские компоненты:** Компоненты с `'use client'` рендерятся на клиенте, как в SPA. Их код попадает в JS-бандлы.
  - **Умный чанкинг:** Next.js теперь автоматически разделяет бандл не только по страницам (как в Pages Router), но и по клиентским компонентам.
- [ ] Next.js не требует знать всё и  сразу. Вы можете начинать с очень простого уровня, и ваше первоначальное понимание — это идеальный старт:
  - **Уровень 1 (Базовый):** Делайте страницы в `app/about/page.js`, используйте `<Link>` и `<Image>`. Данные получайте через `fetch` в серверных компонентах. Этого хватит для 80% сайтов.
  - **Уровень 2 (Продвинутый):** Начните разделять компоненты на серверные и клиентские (`'use client'`). Используйте `generateStaticParams` для SSG. Настройте немного API роуты.
  - **Уровень 3 (Экспертный):** Глубже вникайте в кэширование данных, middleware, инъекцию стилей, кастомные конфиги сборки, оптимизацию бандлов.
